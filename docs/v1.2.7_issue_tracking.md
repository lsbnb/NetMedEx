## V127-01 Language Issue in Search

Symptoms:
- Non-English Search queries may not produce the expected PubTator query/results.
- Search language handling may differ between AI Search enabled and disabled.

Current findings:
- Search language detection is implemented with Unicode-range heuristics in `webapp/callbacks/pipeline.py::detect_query_language`.
- AI Search path pre-translates non-English queries to English, then calls `llm_client.translate_query_to_boolean`.
- Non-AI Search path only translates non-English queries when `llm_client.client` exists; otherwise the original non-English query is sent to PubTator.

Relevant code:
- `webapp/callbacks/pipeline.py:42` language detection.
- `webapp/callbacks/pipeline.py:321` AI Search pre-translation.
- `webapp/callbacks/pipeline.py:367` non-AI Search translation gate.

Risks / suspected causes:
- If LLM is not configured, CJK queries may reach PubTator untranslated and return fewer/no articles.
- The detector labels all CJK ideographs as Traditional Chinese; Japanese Kanji-only queries can be misclassified.
- Boolean translation failures fall back to the current query, which may already be translated or may still be original depending on failure point.

Next steps:
- Add regression tests for Traditional Chinese, Simplified Chinese, Japanese Kana, Japanese Kanji-only, Korean, and English queries.
- Improve Japanese Kanji-only vs Chinese detection if needed.

Fix notes:
- Non-English Search now requires an active LLM instead of silently sending untranslated CJK/Korean text to PubTator.
- Final PubTator query is logged after translation/AI Search processing.

## V127-02 LLM Setting

Symptoms:
- Advanced Settings may show one provider/model while background search/chat uses another.
- NVIDIA NIM appears in UI but may not be propagated into all long-running callbacks.

Current findings:
- Advanced Settings includes NVIDIA NIM fields and persistence.
- `webapp/llm.py` supports provider `nvidia`.
- Search pipeline callback currently receives only OpenAI, Google, Local, and OpenRouter fields. It lacks NVIDIA state inputs and falls through to the local provider branch when `llm_provider == "nvidia"`.
- Manual Chat initialization has the same provider fall-through risk because it also lacks NVIDIA state inputs.

Relevant code:
- `webapp/components/advanced_settings.py:89` provider selector includes NVIDIA.
- `webapp/llm.py:104` NVIDIA env/client support.
- `webapp/callbacks/llm_callbacks.py:441` verify connection handles NVIDIA.
- `webapp/callbacks/pipeline.py:232` search pipeline LLM initialization lacks NVIDIA branch.
- `webapp/callbacks/chat_callbacks.py:641` manual chat LLM initialization lacks NVIDIA branch.

Risks / suspected causes:
- Selecting NVIDIA in Advanced Settings can silently initialize Local Ollama in Search/Chat callbacks.
- Saved settings and verified connection can diverge from actual callback execution.

Next steps:
- Verify Search and Chat with NVIDIA NIM from UI settings.
- Add tests for provider/model resolution across OpenAI, Google, OpenRouter, NVIDIA, and Local.

Fix notes:
- Added shared LLM initialization helper in `webapp/llm.py`.
- Search, auto Chat, manual Chat, and restart Chat rebuild now use the same provider dispatch.
- NVIDIA NIM state is now passed into Search/Chat callbacks instead of falling through to Local.
- `LLMClient.initialize_client()` now clears stale clients when a provider lacks a valid key.

## V127-03 Network Graph Lag Seriously

Symptoms:
- Network Graph interaction or filtering is very slow on larger graphs.

Current findings:
- Graph update callback rebuilds graph elements when filters, layout, node degree, edge cutoff, or community settings change.
- Rebuild uses `rebuild_graph(..., with_layout=True)` followed by `create_cytoscape_js`, then sends all nodes/edges to Dash Cytoscape.
- Default layout is fCose, which can be expensive for large graphs.

Relevant code:
- `webapp/callbacks/graph_update.py:211` rebuild/layout change decision.
- `webapp/callbacks/graph_update.py:260` backend graph rebuild.
- `webapp/callbacks/graph_update.py:270` conversion to Cytoscape elements.
- `webapp/callbacks/graph_update.py:278` layout config generation.
- `webapp/components/graph_tools.py:24` default graph layout is `fcose`.

Risks / suspected causes:
- Server-side rebuild and client-side layout may both be happening for the same interaction.
- Full element replacement causes expensive Cytoscape rerenders.
- Edge labels and directional styling may increase render cost on dense graphs.

Next steps:
- ~~Add timing logs around rebuild, Cytoscape JSON conversion, and client layout phases.~~ ✅
- Reproduce with graph sizes: small (<500 nodes), medium (500-1000), large (>1000).
- ~~Consider debouncing sliders and avoiding rebuild when only display-only filters change.~~ ✅

Fix notes:
- Added server-side graph update timing logs.
- Large graphs with >700 visible nodes now use Cytoscape `preset` layout to avoid a second expensive client-side fCoSE pass after server-side layout.
- `node-degree` number input now uses `debounce=True` (`webapp/components/graph_tools.py`) to avoid triggering graph rebuild on every keystroke.

## V127-04 圖形出現緩慢

Symptoms:
- After Search completes, the graph appears slowly or appears blank for a noticeable time.

Current findings:
- Initial graph load sets `is-new-graph=True`, then `update_graph` rebuilds and sends elements to Cytoscape.
- Graph empty/visible state and Cytoscape element delivery happen through separate callbacks/stores, so UI may show a gap between search completion and rendered graph.

Relevant code:
- `webapp/callbacks/pipeline.py` returns `is-new-graph=True` and switches to the Graph tab after data is saved.
- `webapp/callbacks/graph_update.py:240` rebuild/render callback path.
- `webapp/components/graph.py:42` Cytoscape starts with empty elements.

Risks / suspected causes:
- Heavy layout and full graph serialization delay first paint.
- Search pipeline completes before graph render callback finishes.
- Large graphs may need progressive/placeholder rendering rather than waiting for full layout.

Next steps:
- ~~Add UI status while graph elements/layout are still being prepared.~~ ✅
- Benchmark initial graph render separately from PubTator fetch and graph construction.
- Share optimization work with V127-03.

Fix notes:
- Large graph initial render benefits from the same `preset` layout fallback added for V127-03.
- Server timing logs now separate graph rebuild and Cytoscape JSON conversion.
- Cytoscape component is now wrapped in `dcc.Loading` (`webapp/components/graph.py`) to display a spinner while the server-side graph rebuild callback is running.

## V127-05 取得文獻遠低於設定值

Symptoms:
- Retrieved article count can be much lower than the configured Max Articles.

Current findings:
- UI Max Articles slider allows 10-3000 and defaults to 30.
- PubTator search requests page 1 with `limit=100`, then requests subsequent pages with the selected sort.
- Page 1 request does not pass the selected sort parameter, while later pages do.
- Annotation retrieval batches PMIDs in groups of 100. If PubTator export returns fewer records than requested, the current UI progress can still reach the PMIDs requested total.

Relevant code:
- `webapp/components/advanced_settings.py:11` Max Articles slider.
- `webapp/callbacks/pipeline.py:414` `max_articles` passed into PubTatorAPI.
- `netmedex/pubtator.py:198` query search and page calculation.
- `netmedex/pubtator.py:341` page 1 search query lacks sort parameter.
- `netmedex/pubtator.py:365` subsequent page query includes sort parameter.
- `netmedex/pubtator.py:281` annotation batch retrieval.

Risks / suspected causes:
- Mixed sorting between page 1 and later pages can cause duplicated/missing PMIDs across pages.
- PubTator export may omit articles without available annotations/full text, causing final parsed article count to be lower than PMID count.
- Query translation failures can narrow the query unexpectedly.

Next steps:
- Compare requested PMIDs, exported records, parsed articles, and graph `pmid_title` count in logs/UI.
- Add a warning when final parsed articles are significantly below `max_articles`.

Fix notes:
- Page 1 PubTator search now receives the selected sort, matching later pages.
- PMIDs are deduplicated before annotation fetch while preserving order.
- Logs now warn when parsed annotated articles are lower than requested PMIDs.

## V127-06 Session Expired After Server Restart

Symptoms:
- Chat shows: `Session expired (server was restarted). Please run a new search to rebuild the graph and start chatting.`

Current findings:
- Chat sessions are stored only in the process-local `_sessions` dict, keyed by graph path.
- Graph files remain on disk through the signed `current-session-path`, but Chat does not reconstruct `_sessions` from disk after restart.

Relevant code:
- `webapp/callbacks/chat_callbacks.py:13` process-local `_sessions`.
- `webapp/utils.py:102` signed browser session path can recover graph file location.
- `webapp/callbacks/chat_callbacks.py:1118` missing `_sessions` entry returns the session expired message.

Root cause:
- Expected behavior with current architecture: server restart clears memory-only ChatSession/RAG state.

Next steps:
- Keep the current warning only when the signed session path is invalid or graph file is missing.

Fix notes:
- Send-message callback now attempts lazy ChatSession reconstruction from the persisted `G.pkl` when `_sessions` was cleared by server restart.
- The same Advanced Settings LLM state is passed into send-message so UI-only API keys can be reused for rebuild.

## V127-07 Graph Data Cannot Be Indexed / Graph Panel Data Not Carried to Chat Panel

Symptoms:
- Sometimes Chat reports that graph data cannot be indexed.
- Selected Graph Panel data does not reliably become Chat Panel indexing context.

Current findings:
- Manual Chat indexing uses `cy.selectedNodeData` and `cy.selectedEdgeData`.
- PMIDs are collected from selected edge/node `pmids` fields, then matched against `G.graph["pmid_abstract"]`.
- Large graphs (>1000 nodes) only index selected core nodes into NodeRAG for speed.
- Core node mapping relies on selected node labels and selected edge `source_name` / `target_name` fields.

Relevant code:
- `webapp/callbacks/chat_callbacks.py:558` selected graph data passed into manual Chat initialization.
- `webapp/callbacks/chat_callbacks.py:719` PMIDs collected from selected edges.
- `webapp/callbacks/chat_callbacks.py:731` PMIDs collected from selected nodes.
- `webapp/callbacks/chat_callbacks.py:795` no documents fallback.
- `webapp/callbacks/chat_callbacks.py:830` label-to-raw-node mapping.
- `webapp/callbacks/chat_callbacks.py:851` partial NodeRAG indexing for large graphs.

Risks / suspected causes:
- Cytoscape selected data may not include `pmids`, `source_name`, or `target_name` consistently after graph rebuild/filter/layout changes.
- Label-based raw-node matching can fail if labels are not unique, are truncated, or differ from graph node `name`.
- Partial NodeRAG indexing can make follow-up graph search miss unindexed nodes in large graphs.
- There is no preflight diagnostic showing selected nodes/edges, PMID count, abstract match count, and node index count before indexing.

Next steps:
- ~~Add a preflight validation report before Chat indexing.~~ ✅
- ~~Track partial vs full NodeRAG indexing in session metadata and user-visible status.~~ ✅

Fix notes:
- Cytoscape node/edge data now includes stable raw graph IDs.
- Manual Chat indexing now falls back to persisted NetworkX graph PMIDs when selected Cytoscape payloads lack `pmids`.
- Large-graph NodeRAG partial indexing now uses raw IDs when available before falling back to label matching.
- Preflight diagnostic now logs `selected_nodes`, `selected_edges`, `pmids`, `abstracts_matched/total` before indexing; warns when PMIDs have no matching abstract text (`webapp/callbacks/chat_callbacks.py`).
- Indexing summary (`📊 N abstracts · M nodes (mode)`) is shown in `chat-analyze-status-text` after successful initialization; mode is `partial`, `full`, or `cached`.
